# 한글화 및 조선식 세계관 개편 계획

## 1. 문서 목적

이 문서는 `edict-korea-chosen` 저장소를 대상으로 진행할 한글화 및 조선식 세계관 개편의 기준 문서다.

이번 1차 개편의 목적은 다음과 같다.

- 사용자에게 노출되는 중국어 중심 UI와 문서를 한국어로 통일한다.
- 기존 `삼성육부` 계열 세계관을 조선식 조직 체계로 재해석한다.
- 내부 식별자와 런타임 호환성을 유지하면서 표시 계층만 안정적으로 교체한다.
- 이후 실제 구현과 문서 수정이 흔들리지 않도록 명칭, 범위, 비범위, 검수 기준을 먼저 고정한다.

## 2. 개편 원칙

### 2.1 사용자 노출면 우선

1차 개편은 사용자에게 직접 보이는 요소를 우선 정리한다.

- 프론트엔드 탭명, 버튼, 상태 라벨, 설명문
- 서버 기본 응답 문구 및 기본 표시값
- 에이전트 역할 설명, 예시 발화, 데모 데이터
- 제품 소개 문서, 시작 가이드, 구조 문서, 예시 문서

### 2.2 내부 식별자 유지

다음 항목은 1차 개편에서 유지한다.

- agent id: `taizi`, `zhongshu`, `menxia`, `shangshu`, `hubu`, `libu`, `bingbu`, `xingbu`, `gongbu`, `libu_hr`, `zaochao`
- 상태 키: `Pending`, `Taizi`, `Zhongshu`, `Menxia`, `Assigned`, `Next`, `Doing`, `Review`, `Done`, `Blocked`, `Cancelled`
- 디렉터리명, 워크스페이스 경로명, JSON 키, 테스트의 내부 식별값

이 원칙을 유지하는 이유는 다음과 같다.

- OpenClaw 워크스페이스 경로 및 agent 세션 키와 직접 연결되어 있다.
- `scripts/kanban_update.py`, `dashboard/server.py`, `dashboard/court_discuss.py`, 프론트엔드 상태 매핑, 테스트 코드가 같은 키를 공유한다.
- 데모 데이터와 실제 런타임 데이터가 내부 키 기준으로 누적되어 있다.

즉, 이번 작업은 `식별자 마이그레이션`이 아니라 `표시 계층 개편`이다.

### 2.3 조선식 명칭은 기능 우선

조선의 실제 `3사6조` 체계와 현재 시스템의 기능은 1:1 대응이 아니다.

따라서 이번 개편에서는 역사적 엄밀성보다 현재 시스템의 역할 구조를 설명하기 쉬운 방향으로 조선식 명칭을 재해석한다.

- 기획 기능은 조선식 언관/문한 체계에 맞게 재해석한다.
- 심의 기능은 견제와 간쟁 기능 중심으로 재구성한다.
- 배분/지시 기능은 승정원 계열 보조 체계로 설명한다.
- 아침 브리핑 역할은 조보/조보청 계열로 설명한다.

## 3. 목표 구조

### 3.1 설명 방식

문서와 UI에서는 다음 방식으로 설명한다.

- 핵심 구조: `3사6조 기반 운영 체계`
- 보조 역할: `세자`, `승정원`, `조보 역할`

즉, 표면 설명은 `3사6조`를 중심으로 하되 실제 시스템에는 추가 허브 역할이 있음을 명시한다.

### 3.2 표시 명칭 가이드

아래 표는 1차 개편에서 사용할 권장 표시 명칭이다.

| 내부 ID | 현재 역할 성격 | 1차 표시 명칭 | 설명 원칙 |
|---|---|---|---|
| `taizi` | 입력 분류, 초동 정리 | 세자 | 상위 지시 수신, 초동 분류, 간단 응답 |
| `zhongshu` | 기획, 초안, 분해 | 홍문관 | 지시 해석, 실행안 기획, 작업 분해 |
| `menxia` | 심의, 반려, 검토 | 사간원 | 계획 검토, 위험 지적, 반려/수정 요구 |
| `shangshu` | 배분, 조율, 취합 | 승정원 | 승인된 안건 배분, 집행 조율, 결과 취합 |
| `hubu` | 자원, 데이터, 비용 | 호조 | 데이터, 자원, 수치 분석 |
| `libu` | 문서, 형식, 보고 | 예조 | 문서 작성, 보고 정리, 기준 정비 |
| `bingbu` | 구현, 코드, 기술 집행 | 병조 | 기능 구현, 코드 작업, 기술 대응 |
| `xingbu` | 감사, 보안, 규정 점검 | 형조 | 보안, 규정, 검토, 통제 |

- `zhongshu`를 실제 조선 관제의 특정 기관으로 단정하지 않는다.
- `menxia`도 역사적 동일기관 복원으로 설명하지 않는다.
- `shangshu`는 현재 시스템상 배분 허브 역할이므로 `승정원`으로 재표현한다.
- `zaochao`는 기존 이름이 기능명에 가까우므로 `조보청` 또는 `조보` 계열로 설명한다.

## 4. 상태 및 화면 표기 원칙

### 4.1 상태 키와 표시명 분리

내부 상태 키는 유지하고, 화면에는 한국어 표시명을 사용한다.

예시:

| 내부 상태 키 | 화면 표시명 초안 |
|---|---|
| `Pending` | 접수 대기 |
| `Taizi` | 세자 분류 |
| `Zhongshu` | 홍문관 기안 |
| `Menxia` | 사간원 심의 |
| `Assigned` | 승정원 배분 완료 |
| `Next` | 집행 대기 |
| `Doing` | 집행 중 |
| `Review` | 취합 검토 |
| `Done` | 완료 |
| `Blocked` | 중단 |
| `Cancelled` | 취소 |

### 4.2 파이프라인 설명 문구

기존 파이프라인이 `황상 → 태자 → 중서 → 문하 → 상서 → 육부` 식으로 보이던 부분은 아래 방향으로 정리한다.

예시 문구:

```text
임금의 지시 → 세자 분류 → 홍문관 기안 → 사간원 심의 → 승정원 배분 → 육조 집행 → 결과 상신
```

단, 코드 내부의 상태 전이 키는 건드리지 않는다.

## 7. 검수 기준

### 7.1 문자열 검수

다음 조건을 만족해야 한다.

- 1차 대상 파일군에서 중국어 중심 표시 문자열이 의도치 않게 남아 있지 않다.
- 같은 역할이 문서마다 다른 한글 명칭으로 불리지 않는다.
- 내부 ID와 표시명이 섞여 보여도 사용자가 혼란스럽지 않도록 설명이 있다.

### 7.2 호환성 검수

다음 조건을 유지해야 한다.

- 내부 agent id 기반 필터와 라우팅이 그대로 동작한다.
- 상태 전이 로직이 깨지지 않는다.
- 데모 데이터가 화면에서 정상 렌더링된다.
- 세션명, task id, `agent:<id>:...` 패턴 등 내부 키 기반 표시가 망가지지 않는다.

### 7.3 문서 일관성 검수

다음 문서 간 설명이 서로 충돌하지 않아야 한다.

- `README.md`
- `docs/getting-started.md`
- `docs/task-dispatch-architecture.md`
- `agents/GLOBAL.md`
- 각 `SOUL.md`

## 8. 후속 2차 작업 범위

다음 작업은 2차로 분리한다.

- 내부 agent id 자체를 조선식 식별자로 바꾸는 작업
- 상태 키 자체를 한국어 키로 바꾸는 작업
- 디렉터리/워크스페이스/세션 키 마이그레이션
- 기존 JSON 데이터 자동 변환기 작성
- 테스트 코드와 마이그레이션 코드 전면 개편
- `scripts/kanban_update.py` 의 사용자 노출 출력 문자열 한글화
  - state, flow, progress, done 등 모든 명령의 stdout/stderr 메시지
  - remark/comment 등 칸반에 표시되는 필드 처리 로직의 문구
  - 사용자 라벨 정규화 로직 (예: `传旨/下旨` 접두사 제거 후 한국어 보존)
- `scripts/skill_manager.py` 의 사용자 노출 출력 문자열 한글화
  - add-remote, list-remote, update-remote, import-official-hub 등 명령의 출력 메시지
  - 오류/성공/안내 문구
- 위 두 파일 외 `./*.sh`, `scripts/*.py`, `scripts/*.sh`, `dashboard/auth.py` 의 코드 주석 한글화 (선택, 가독성 향상 목적)
- **한자어 현대 한국어 친화 재번역** (8.1 항목 참고)

이 작업은 별도 설계가 필요하다. 특히 아래 항목을 먼저 정리해야 한다.

- 기존 ID와 새 ID의 양방향 호환 규칙
- 저장 데이터 자동 마이그레이션 방식
- OpenClaw 외부 런타임과의 경로 호환
- 기존 문서와 데모 데이터의 변환 범위

### 8.1 한자어 현대 한국어 재번역 작업

1차 작업에서 한자 병기 (`한국어(한자)`) 형태로 보존했거나 단독 한자어로 남긴 표현 중, 현대 한국어 사용자가 의미를 직관적으로 파악하기 어려운 항목을 친화적인 표현으로 재번역한다.

#### 8.1.1 적용 원칙

- 외래 서비스명은 유지: `Feishu(飞书)`, `WeCom(企业微信)`
- 직위명·기관명은 유지: `세자`, `홍문관`, `사간원`, `승정원`, 6조 명칭, `관상감`, `조보청`
- 그 외 행위·상태·결과를 가리키는 한자어는 현대어로 치환한다
- 의미 명확화를 위해 문맥에 따라 부수 단어 추가를 허용한다 (예: `下旨` → 단순 "어명" 보다 "어명 하달" 이 더 자연스러운 곳)

#### 8.1.2 변환 매핑 표

##### 어명·결재·보고 계열 (한자 병기 → 현대어)

| 1차 표기 | 위치 수 | 2차 변환 |
|---|---|---|
| `하지(下旨)` | 10 | 어명 하달 |
| `전지(傳旨)` | 2 | 어명 전달 |
| `어비(御批)` | 1 | 임금 결재 |
| `전보(轉報)` | 1 | 보고 전달 |
| `지시(旨意)` | 1 | 지시 |
| `유전(流轉)` | 1 | 흐름 |
| `천하요문(天下要闻)` | 1 | 주요 뉴스 |

##### `agents/*/SOUL.md` 직무 설명의 한자어

| 위치 | 1차 표기 | 2차 변환 |
|---|---|---|
| `agents/libu/SOUL.md` | 전장(典章)과 의제(儀制) | 법령과 의례 제도 |

#### 8.1.4 검수 기준

- 위 매핑 표의 모든 항목이 일관되게 적용되었다.
- 같은 단어가 한 곳은 옛 표기, 다른 곳은 새 표기로 불리지 않는다.
- UI 표기가 변경된 곳은 스크린샷 갱신 여부를 별도 기록한다 (1차에서는 스크린샷 재촬영 제외 항목이었음).
- 직위명·기관명은 변경되지 않았다.
